# PROMPT_95: Precomputed Profile Auto-Odyssey (Remove Runtime Recursive CTE)

## Objective

Language profile pages currently render a "Dynamic Heritage Journey" section by
calling `DataLoader.get_auto_odyssey()` from the `/language/{name}` route. That
method runs a recursive CTE at request time. Earlier graph-intelligence prompts
introduced closure tables and generated-data artifacts so profile pages can
render graph-derived enrichments without doing recursive graph traversal during
normal page requests.

This prompt replaces the profile page's runtime auto-Odyssey query with the
pre-generated artifact:

```text
generated-data/odyssey/auto-odyssey-candidates.json
```

The live backend should read this generated JSON when rendering profiles. The
static site build should bake the resulting journey cards directly into the
generated HTML during `make site`; GitHub Pages does not need to copy or fetch
`generated-data/` at runtime for this feature.

Important schema note: as of PROMPT_94, this artifact may only contain a
top-level list of source-language candidates with aggregate descendant counts.
That is not enough to render the existing profile journey cards. This prompt
therefore includes the small derived-data schema expansion needed to store the
precomputed descendant steps for each source language.

This prompt does **not** remove `DataLoader.get_auto_odyssey()` entirely. Keep
it available for explicit dynamic use by CLI/TUI/future tooling unless a later
prompt chooses to deprecate it.

## Product Requirements

### 0. Expand the generated auto-Odyssey artifact

Before changing the profile route, inspect the current artifact schema. Update
`scripts/generate_derived_data.py` so `build_auto_odyssey_candidates()` emits
per-source journey steps.

The artifact must remain a single JSON file at:

```text
generated-data/odyssey/auto-odyssey-candidates.json
```

Required shape:

```json
{
  "generated_at": "...",
  "count": 20,
  "candidates": [
    {
      "name": "Python",
      "display_name": "Python",
      "year": 1991,
      "direct_descendant_count": 4,
      "total_descendant_count": 12,
      "steps": [
        {
          "language": "TypeScript",
          "display_name": "TypeScript",
          "milestone": "Generation 1 Impact"
        }
      ]
    }
  ]
}
```

Implementation rules:

- Keep the existing aggregate fields (`direct_descendant_count`,
  `total_descendant_count`) for compatibility with existing tests and reports.
- Add `steps` for each candidate source language.
- Compute `steps` at derived-data generation time using closure tables or direct
  influence tables. Do not leave the profile route to compute descendants.
- Prefer the same ranking intent as `DataLoader.get_auto_odyssey()`: influential
  descendant languages, limited to 4 steps, with deterministic ordering.
- Include only `entity_type = 'language'` descendants in journey steps unless the
  existing dynamic behavior explicitly includes mixed language-like entities.
- If the current top-20 source-language query excludes common profile test
  targets such as Python, do not force Python into production output just for a
  test. Test fixtures can provide a minimal generated artifact for Python.

### 1. Stop profile pages from calling `DataLoader.get_auto_odyssey()`

In `src/app/app.py`, the `/language/{name}` route must no longer call:

```python
data_loader.get_auto_odyssey(...)
```

for normal profile rendering.

Instead, the route should obtain profile journey data from a new helper that
reads the generated Odyssey candidates artifact.

The route should still pass an `auto_odyssey` object into `profile.html`, so
the template contract remains stable.

This prompt must implement the full generated-data flow in one pass:

- Update `scripts/generate_derived_data.py` to produce journey `steps`.
- Add or update tests for the generated artifact schema.
- Add the generated-data route helper in `src/app/app.py`.
- Replace the `/language/{name}` route's implicit dynamic call with that helper.
- Update `src/app/templates/profile.html` to prefer `step.display_name` when
  present while preserving `step.language` routes.
- Add a static export assertion that generated journey cards are baked into
  profile HTML.
- Update the required docs.

### 2. Add a generated-data helper for profile auto-Odyssey

Add a helper in `app.py`, near the existing generated-data/report helpers:

```python
def _load_profile_auto_odyssey(language_name: str) -> Optional[Dict[str, Any]]:
    ...
```

Behavior:

- Read `generated-data/odyssey/auto-odyssey-candidates.json` using the existing
  generated-data loading helper pattern.
- Return `None` if the file is absent, malformed, missing the language, or the
  language has no usable candidates.
- Match language by canonical `name` and, if the artifact exposes it, by
  `display_name`. Matching should be case-insensitive.
- Normalize underscores in route names the same way the rest of the profile
  route does: `Python` and `Python`, `Common_Lisp` and `Common Lisp` should be
  treated as equivalent where practical.
- Return an object shaped for the existing profile template:

```python
{
    "id": "auto_python",
    "title": "The Legacy of Python",
    "description": "A precomputed journey through influential descendants in the Python lineage.",
    "steps": [
        {
            "language": "TypeScript",
            "display_name": "TypeScript",
            "milestone": "Influential Descendant",
        }
    ],
}
```

The helper should read the `steps` emitted by the generated artifact. If the
artifact exists but a matching candidate lacks `steps`, return `None` rather
than running a fallback query.

### 3. Preserve graceful degradation

If `generated-data/odyssey/auto-odyssey-candidates.json` does not exist, the
profile page must still render successfully.

In that case:

- `auto_odyssey` should be `None`
- The "Dynamic Heritage Journey" section should not render
- No request-time recursive fallback should run from the profile route
- No 500 should occur

This is important because a developer may run the server after `make build` but
before `make derived-data`.

### 4. Keep explicit dynamic auto-Odyssey behavior available elsewhere

Do not delete or rewrite `DataLoader.get_auto_odyssey()` in this prompt unless a
small compatibility adjustment is required by tests.

This prompt is only about removing the implicit recursive query from profile
page rendering. If CLI/TUI or other explicit surfaces call `get_auto_odyssey()`,
leave those surfaces alone.

### 5. Static site behavior

The static site generated for GitHub Pages should not need runtime access to
`generated-data/odyssey/auto-odyssey-candidates.json`.

Expected flow:

```bash
make build
make derived-data
make site
```

During `make site`, `SiteCrawler` fetches rendered profile pages from the live
FastAPI app. The app reads the generated Odyssey candidates file and renders the
journey section into normal HTML. The static export writes that HTML to
`site/language/.../index.html`.

Therefore:

- Do not add browser-side `fetch()` calls for this feature.
- Do not require copying `generated-data/` into `site/`.
- Do not add a new static asset route for the generated Odyssey JSON.
- Do verify the crawled HTML contains the generated journey section when the
  artifact exists.

### 6. Documentation updates

Update `src/README.md`:

- Note that language profile pages use precomputed generated-data for
  auto-Odyssey journey cards.
- Document that `make derived-data` should be run before `make site` when
  generating static pages with graph-derived enrichments.
- Clarify that generated Odyssey data is baked into static HTML during
  `make site`; GitHub Pages does not need to serve `generated-data/` for this
  feature.

Update `.context/routes.md`:

- Add a note to the `/language/{name}` row that the Dynamic Heritage Journey is
  sourced from generated-data rather than a request-time recursive CTE.

Update `.context/data-pipeline.md` if it does not already mention:

- `generated-data/odyssey/auto-odyssey-candidates.json`
- The dependency order `make build` → `make derived-data` → `make site`
  for static profile enrichments.

## Implementation Guidance

### Inspect the artifact schema first

Before implementing, inspect both:

```bash
sed -n '1,260p' scripts/generate_derived_data.py
test -f generated-data/odyssey/auto-odyssey-candidates.json && \
  sed -n '1,160p' generated-data/odyssey/auto-odyssey-candidates.json
```

If the generated file is absent, run:

```bash
make derived-data
```

If the generated file does not contain per-language `steps`, update the
derived-data generator as described in Requirement 0. Keep the generator change
small and add tests for the expanded schema.

### Suggested helper shape

The exact implementation should follow the actual artifact schema, but this is
the intended direction:

```python
def _normalize_generated_name(name: str) -> str:
    return " ".join(name.replace("_", " ").split()).lower()

def _load_profile_auto_odyssey(language_name: str) -> Optional[Dict[str, Any]]:
    data = _load_generated_data("odyssey/auto-odyssey-candidates.json")
    if not data:
        return None

    # Locate language record in the artifact by canonical name/display name.
    # Validate and copy up to 4 precomputed steps from the artifact.
    # Return the existing template contract with id/title/description/steps.
```

If the artifact stores all candidates in a list, build a small lookup lazily and
cache it by `GENERATED_DATA_DIR` so tests can monkeypatch `GENERATED_DATA_DIR`
without stale results.

If the artifact stores a top-level mapping, direct lookup is fine.

### Avoid stale cache bugs in tests

Existing app helpers cache graph report data based on `GENERATED_DATA_DIR`.
Follow the same pattern if you cache auto-Odyssey candidates:

```python
_auto_odyssey_dir: Optional[str] = None
_auto_odyssey_candidates: Dict[str, Dict[str, Any]] = {}
```

When `GENERATED_DATA_DIR` changes, reload from disk.

### Generator query guidance

The current dynamic `DataLoader.get_auto_odyssey()` ranks descendants with a
recursive CTE over `influences`. For the precomputed artifact, prefer closure
tables so generation is deterministic and request-time traversal is eliminated.
A reasonable derived-data query for each source language is:

```sql
SELECT DISTINCT l.name, l.display_name, ld.depth, l.influence_score
FROM language_descendants ld
JOIN languages l ON l.id = ld.descendant_language_id
WHERE ld.root_language_id = ?
  AND COALESCE(l.entity_type, 'language') = 'language'
ORDER BY l.influence_score DESC, ld.depth ASC, l.name ASC
LIMIT 4
```

Map each row to:

```python
{
    "language": row["name"],
    "display_name": row["display_name"] or row["name"],
    "milestone": f"Generation {index + 1} Impact",
}
```

### Profile route change

Current pattern likely resembles:

```python
auto_odyssey = (
    data_loader.get_auto_odyssey(lang["name"])
    if entity_type == "language"
    else None
)
```

Replace it with:

```python
auto_odyssey = (
    _load_profile_auto_odyssey(lang["name"])
    if entity_type == "language"
    else None
)
```

Do not add a fallback to `data_loader.get_auto_odyssey()` from this route. That
would preserve the runtime recursive CTE and defeat the point of the prompt.

### Template compatibility

The existing template expects:

```jinja2
auto_odyssey.description
auto_odyssey.steps
step.language
step.milestone
```

Keep those keys present. If you add `display_name`, update the template to
prefer `step.display_name or step.language` while preserving existing behavior.
Links must still use `step.language` so route behavior remains stable even when
display text changes.

```json

      "name": "Python",
```

Create only the fields the helper reads. A minimal fixture should include
`candidates[0].steps`, not only aggregate counts.
