# PROMPT_74: Concepts and Paradigms — Reactive UI, State Flow, and Host-Language DSLs

## Objective

Expand the Atlas's concept and paradigm coverage for the ideas behind modern
reactive UI systems, especially the React, Redux, Elm, SwiftUI, and Svelte
lineages. This batch should capture the historical shift from imperative DOM
and widget mutation toward component models, state flow, host-language UI
DSLs, and compilation-first interface systems.

2. Load `.context/data-pipeline.md`.

10. Check off each item immediately after completing it.

## Canonicalization Rule

Dark matter in this area often appears as architectural claims rather than
clean noun phrases. Canonicalize to standard concept names, but explicitly
absorb the phrasing that already appears in Atlas narratives.

Examples for this batch:

- `The component model gave frontend teams a durable unit for composition`
  -> `Component Model`
- `Reducers turned state change into a pure function` -> `Reducers`
- `Protocols with default implementations` should be explained inside
  `Protocol-Oriented Design`

## Concepts to Add

Add these records to `data/concepts.json` and create one concept profile JSON
file per entry in `data/docs/concept_profiles/`.

1. **Virtual DOM** (`Virtual_DOM.json`)
   - Connect React's reconciliation model to the post-jQuery architecture shift

2. **Component Model** (`Component_Model.json`)
   - Connect Smalltalk/UI composition heritage to React and modern frontend UI

3. **Unidirectional Data Flow** (`Unidirectional_Data_Flow.json`)
   - Connect Elm, Flux, Redux, and predictable interface state

4. **Reducers** (`Reducers.json`)
   - Treat reducers as the explicit state-transition function pattern, not
     just a Redux API detail

5. **Direct Manipulation UI** (`Direct_Manipulation_UI.json`)
   - Connect GUI traditions to declarative reactions against imperative widget
     plumbing

6. **State-Based Transitions** (`State_Based_Transitions.json`)
   - Connect finite-state reasoning, TEA, and modern view recomposition

7. **Protocol-Oriented Design** (`Protocol_Oriented_Design.json`)
   - Connect Swift, SwiftUI, protocol composition, and capability-first design

8. **DSL-Style APIs** (`DSL_Style_APIs.json`)
   - Connect JSX, SwiftUI builders, and host-language interface grammars

9. **No Virtual DOM** (`No_Virtual_DOM.json`)
   - Explain the Svelte and compile-time reaction against VDOM runtimes

10. **Live Previews** (`Live_Previews.json`)
    - Connect SwiftUI and modern interface tooling to immediate feedback loops

## Paradigms to Add

Add these paradigm records to `data/paradigms.json` and update affected
languages in `data/languages.json`.

1. **State-driven UI**
   - Candidate languages to update: Elm, React, Redux, SwiftUI, Svelte where
     historically appropriate

2. **UI DSL**
   - Candidate languages to update: React, SwiftUI, Svelte, and other entries
     whose interface model is best understood as a host-language DSL

## Concept Profile Schema

Use the same schema as PROMPT_72:
`title`, `overview`, `historical_context`, `key_aspects`,
`technical_implications`, `legacy`, `ai_assisted_discovery_missions`.

Additional requirements:

- The historical context must explain why these concepts emerged as a
  reaction to jQuery-era DOM mutation, callback sprawl, and GUI boilerplate.
- The technical implications should make clear what becomes easier or harder
  in state-driven UI systems.

## Paradigm Record Requirements

Use the same `data/paradigms.json` record shape already present in the repo.

Additional requirements:

- `State-driven UI` must be written as a real programming paradigm for
  interface construction, not as product-marketing language.
- `UI DSL` must explain host-language embedded interface notation as a
  language-design move, not just as a pretty syntax layer.

## Cross-Reference Requirements

- Connect `Virtual DOM`, `Component Model`, `Unidirectional Data Flow`, and
  `Reducers` across React, Redux, and Elm.
- Connect `Protocol-Oriented Design`, `DSL-Style APIs`, and `Live Previews`
  to Swift and SwiftUI.
- Use `No Virtual DOM` to explain Svelte's compiler-first reaction.
- Make sure `State-Based Transitions` is linked to predictable UI logic and
  not reduced to a generic state-machine buzzword.
