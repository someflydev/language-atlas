### Group 3: Variable shadowing — reuse of `f` across scopes

Two files use `f` as both a loop variable (str or Path) and a file
handle (TextIOWrapper) in the same function scope. mypy infers an
inconsistent type for `f` across assignments and reports errors.

**`src/app/core/data_loader.py` (lines 72-88):**

In `_load_json_dataset`, the inner loop uses `for f in files:` where
`f` is a `str` filename. Later, `with open(f_path, ...) as f:` reuses
`f` as a `TextIOWrapper`. Rename the file-handle alias in the `with`
statement to `fh` throughout that function:

    with open(f_path, 'r', encoding='utf-8') as fh:
        data = json.load(fh)

**`scripts/dark_matter_audit.py`:**

conflicts with an earlier loop variable `f` that mypy has typed as
`Path`. There are at least 9 affected blocks (lines 201, 210, 253, 257,
275, 285, 331, 339, 342, 379 per the error report). Rename the file
handle to `fh` in all `with open(...) as f:` blocks throughout the
entire file. Also update any `json.load(f)` / `json.dump(..., f)` calls
inside those blocks to use `fh`.

After renaming, re-run mypy to confirm all assignment/arg-type errors in

### Group 4: no-any-return — functions returning Any

Several functions in `data_loader.py` and `app.py` are annotated to
return `dict[str, Any] | None`, `str | None`, or `list[dict[str, Any]]`
but return `Any` because `sqlite3.fetchone()` and `dict.get()` return
`Any` when the connection uses `row_factory = sqlite3.Row`.

The cleanest fix per site:

**`src/app/core/data_loader.py`:**

At each `[no-any-return]` site (lines 258, 260, 292, 294, 326, 328,
793, 831, 836, 901, 906), the pattern is a SQLite `fetchone()` result
being returned directly. Wrap each bare `return row` or `return
row["field"]` with an explicit cast:

    from typing import cast
    # already imported — just add cast at each return site

Examples:
- `return cast(dict[str, Any], row)` for `dict[str, Any] | None` returns
- `return cast(str, row["field"])` for `str | None` returns

Alternatively, add a local narrowing variable:

    result: dict[str, Any] | None = dict(row) if row else None
    return result

Choose whichever is least invasive at each site. The goal is no
`[no-any-return]` errors, not uniformity.

**`src/app/app.py`:**

Lines 238, 242, 939, 987, 1001, 1006, 1011, 1016, 1024, 1029, 1037,
1042, 1050, 1055, 1059, 1066, 1070, 1077, 1093 — all are route handlers
returning data from `self.loader.*` calls. The simplest fix here is to
cast at each return site, or to drop the explicit return type annotation
from the route handler if FastAPI does not require it for correctness.

Since FastAPI uses Pydantic/response models, not Python return type
annotations, for validation, the return type annotations on route
handlers are informational only. The pragmatic fix is: for each
affected route function, change the return annotation from the specific
`list[dict[str, Any]]` / `dict[str, Any]` to `Any`, and add
`from typing import Any` if not already imported. This is honest — the
loaders return SQLite rows or JSON, which are dynamically typed.

Do NOT remove return annotations entirely (that would trigger
`no-untyped-def`). Use `-> Any` where the concrete type cannot be
narrowed without significant restructuring.

### Group 7: NoCloseConnection arg-type — test_insights.py

`tests/test_insights.py` defines its own local `NoCloseConnection`
class. `InsightGenerator` and `AtlasAnalytics` accept
`sqlite3.Connection`. Since `NoCloseConnection` uses `__getattr__`
delegation, it is structurally compatible but not nominally a subclass.

Three options (choose the least invasive):

**Option A (recommended):** Make `NoCloseConnection` in
`tests/conftest.py` subclass `sqlite3.Connection` — but that is not
straightforward because `sqlite3.Connection` requires arguments at
construction.

**Option B (recommended):** Change the type signature of
`InsightGenerator.__init__` and `AtlasAnalytics.__init__` in
`src/app/core/insights.py` to accept a structural Protocol instead of
`sqlite3.Connection`. Define a small protocol in `insights.py`:

    from typing import Protocol, Any

    class _Connection(Protocol):
        def execute(self, sql: str, params: Any = ...) -> Any: ...
        def executemany(self, sql: str, seq: Any) -> Any: ...
        def close(self) -> None: ...
        def cursor(self) -> Any: ...

Then type the `conn` parameter as `_Connection`.

**Option C (simplest short-term):** Add `# type: ignore[arg-type]` to
the three constructor call sites in `test_insights.py`. Use this only
if Option B proves too invasive (e.g., it breaks other things).

Also annotate the local `NoCloseConnection` in `test_insights.py` with
proper types (see Group 5 above for the signature to use).
