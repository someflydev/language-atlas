## Background
PROMPT_49 promoted doc generation into `src/app/core/site_builder.py`.
PROMPT_50 added `SiteCrawler`, which spins up the FastAPI app via
`TestClient` and writes a fully-rendered static HTML mirror of the live
site into `site/` (gitignored). The static site currently shows an
unfiltered, inert language grid because there is no server to handle the
HTMX endpoints.

This prompt closes the loop. It adds a small client-side SQLite layer
using **phiresky/sql.js-httpvfs**, which loads the project's SQLite file
in the browser via HTTP range requests, runs queries against it, and
re-injects results into the page. That technique is the one referenced in
the famous "Hosting SQLite databases on GitHub Pages" blog post; it is the
exact pattern this project wants to mirror.

This prompt also lays down the deployment workflow for the long-lived
`gh-pages` branch (the user maintains it manually; this prompt does NOT
create, check out, or push that branch), creates the root-level
`index.html` redirect, and stubs a front-facing `README.md` on `main` that
will become the project's GitHub-first impression.

## Decisions ratified by this prompt

1. **Client-side query engine: phiresky/sql.js-httpvfs.** Loaded from a CDN
   (`jsdelivr` or `unpkg`) so there is no npm/Node tooling in the project.
   The chunked SQLite file plus its `config.json` live under `site/db/`.

   Alternatives considered: the official `@sqlite.org/sqlite-wasm` build,
   and plain `sql.js`. Both require downloading the entire SQLite file
   up-front, which negates the GitHub-Pages-friendly streaming model.
   `sql.js-httpvfs` uses HTTP range requests to fetch only the chunks
   needed for the current query, which keeps page loads fast even for a
   multi-megabyte database.

2. **The `gh-pages` branch is maintained by hand, not by CI.** The user
   has explicitly chosen the manual workflow: check out `gh-pages`, merge
   from `main`, run `make site` and `make pages`, commit, push. This
   prompt provides the tooling and the documentation; it does NOT create
   the branch, push to it, or set up any GitHub Actions workflow.

3. **Root `index.html` is a meta-refresh redirect to `site/index.html`.**
   It exists only on the `gh-pages` branch. Its sole job is to satisfy
   GitHub Pages' "the repo root must have a homepage" requirement and
   forward the visitor into the real site. Keeping it minimal avoids
   maintaining two homepages.

4. **A new front-facing `README.md` lands on `main`.** This is separate
   from `src/README.md` (which is the authoritative codebase guide
   protected by AGENT.md). The new root `README.md` is short, friendly,
   and links into `src/README.md` and `API_GUIDE.md` for depth. It exists
   to be the first thing a visitor sees on github.com.

5. **`make pages` is the one command the user runs on the `gh-pages`
   branch** to produce a deployable tree. It rebuilds the database,
   runs the static site builder, chunks the database for HTTP range
   requests, copies the chunks under `site/db/`, writes `.nojekyll`, and
   writes the root redirect `index.html`. It refuses to run if the
   current branch is `main`.

## Tasks

### Task 1: Add `scripts/prep_pages.py`
Create a Python script (not bash; Python is the project's language and is
already on the path) that:

1. Verifies the current branch is NOT `main`. If it is, exit non-zero with
   a clear message: "Refusing to build pages artifacts on main; switch to
   gh-pages first." Use `subprocess.run(["git", "rev-parse",
   "--abbrev-ref", "HEAD"])` to check.
2. Verifies `language_atlas.sqlite` exists at the repo root. If not,
   instructs the user to run `make build` and exits non-zero.
3. Verifies `site/` exists. If not, instructs the user to run `make site`
   and exits non-zero.
4. Splits `language_atlas.sqlite` into fixed-size chunks suitable for
   `sql.js-httpvfs`. Use a chunk size of 1 MiB (1048576 bytes). Write the
   chunks under `site/db/atlas/` named `db.sqlite3.<n>` (matching the
   `sql.js-httpvfs` "split file" mode), plus a `config.json` next to them
   matching the schema documented at
   https://github.com/phiresky/sql.js-httpvfs#usage. The script should
   compute the file's total length and the request chunk size and emit:
   ```json
   {
     "serverMode": "full",
     "requestChunkSize": 1024,
     "databaseLengthBytes": <length>,
     "urlPrefix": "atlas/"
   }
   ```
   Refer to the upstream README before finalizing the field names; the
   project must match whatever the published version expects.

   If matching upstream's exact split format proves fiddly, the simpler
   fallback is `serverMode: "full"` with the entire `.sqlite` file served
   as a single asset (no chunking). Document whichever approach you
   choose in `docs/GH_PAGES.md`. Streaming is preferred but not required
   for this prompt to be considered complete.
5. Writes `.nojekyll` at the repo root.
6. Writes `index.html` at the repo root with this exact content:
   ```html
   <!doctype html>
   <html lang="en">
     <head>
       <meta charset="utf-8">
       <meta http-equiv="refresh" content="0; url=site/">
       <title>Language Atlas</title>
       <link rel="canonical" href="site/">
     </head>
     <body>
       <p>Redirecting to <a href="site/">Language Atlas</a>.</p>
     </body>
   </html>
   ```
7. Prints a summary: file counts, total bytes, next-step git commands the
   user should run.

### Task 3: Wire `sql.js-httpvfs` into the templates
In `src/app/templates/base.html` (or whichever template is the layout
root), add a conditional block that loads `sql.js-httpvfs` from a CDN and
the project's own `atlas-static.js` shim, gated on a Jinja flag
`atlas_static_mode` (which the crawler in PROMPT_50 already sets via the
environment variable).

```html
{% if atlas_static_mode %}
  <script type="module" src="/static/atlas-static.js"></script>
{% endif %}
```

The script tag must use a relative URL after PROMPT_50's link rewriter
processes it.

Create `src/app/static/atlas-static.js`. It must:

1. Import `createDbWorker` from
   `https://cdn.jsdelivr.net/npm/sql.js-httpvfs@latest/dist/index.js` (or
   pin a specific version; pinning is preferred). If ESM-from-CDN is too
   fragile, fall back to the UMD build and document the choice.
2. On `DOMContentLoaded`, instantiate the worker pointing at
   `db/atlas/config.json` (relative to the page; resolve correctly from
   nested pages by using a `<base>` tag or by computing the depth).
3. Hook the homepage filter form: when the user changes cluster, paradigm,
   or year, run a SQL query against the worker
   (`SELECT * FROM languages WHERE ...`) and rebuild the language grid
   client-side. Reuse the same DOM structure the FastAPI partial emits so
   the visual result matches.
4. Hook the search box: when the user types (debounced 200ms), query the
   FTS5 `search_index` virtual table and render results into the same
   container the live `/search` page uses.
5. Remove the static-mode notice div that PROMPT_50 added once the worker
   has booted successfully.
6. Catch and log any worker errors; on failure, leave the static-mode
   notice in place so the page still displays useful content.

The shim is intentionally small (target: under 200 lines). It is not a
SPA; it is progressive enhancement on top of the already-rendered HTML.

### Task 4: Update PROMPT_50's static-mode notice copy
