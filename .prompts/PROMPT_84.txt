# PROMPT_84: Redesign Paradigm Pages as Foundation-Aware Ecosystem Views

## Objective

PROMPT_83 added a structured paradigm ecosystem model. This prompt uses that
data to turn `/paradigm/{name}` from a generic tag page into a page that shows
both:

- the languages that embody a paradigm
- the foundations that explain where that paradigm comes from

The guiding principle is:

> Paradigms are the primary browsing surface. Foundations are the conceptual
> substrate beneath them.

Do not build a brand-new top-level foundations hub here. The goal is to weave
foundations into paradigm browsing thoughtfully.

## Product Requirements

### 1. Keep the route, change the experience

Retain the existing route:

```text
/paradigm/{name}
```

But stop rendering it through the generic `tag_view.html` pattern.

Create a dedicated paradigm template, for example:

```text
src/app/templates/paradigm_view.html
```

`/cluster/{name}` should continue using the generic tag view. Do not entangle
cluster and paradigm rendering unnecessarily.

### 2. The page must have two clearly distinct sections

The page should include:

1. `Foundational Precursors`
2. `Languages in this paradigm`

Do not mix foundations and languages into a single undifferentiated card grid.

The foundation section should appear before the language grid, or otherwise be
visually higher in hierarchy, so the page reads as:

idea -> embodiment

### 3. Foundation cards should explain why they are present

Each foundation card should surface:

- display name
- year
- short description or philosophy
- a short relevance explanation
- supporting language count
- a few example descendant languages

If the loader exposes `is_directly_tagged`, consider a compact badge such as:

- `Directly tagged`
- `Shared precursor`

Keep it tasteful. Avoid dashboard clutter.

### 4. Language section should preserve existing usefulness

The language list still needs:

- sort by `name` and `year`
- the familiar card-based browse behavior
- links to profiles
- visible paradigm/cluster metadata where helpful

If you create a small shared card partial for language cards, that is fine.
Do not do a large template refactor unless it clearly reduces duplication.

### 5. The page should read like a guided intellectual map

The top of the page should include:

- paradigm title
- description
- compact stats such as:
  - number of languages
  - number of foundations
  - earliest foundation year or earliest language year if available

Optional but encouraged:

- a short line of explanatory copy clarifying that the top section contains
  historical or theoretical precursors while the lower section contains
  executable languages

### 6. Preserve static export compatibility

This route is crawled into the static site. Do not introduce client-only logic
that leaves the exported page empty or broken.

Server-render the full page content.

## Implementation Guidance

### Route logic

Update the `/paradigm/{name}` route in `src/app/app.py` to consume the
paradigm ecosystem data from PROMPT_83.

Keep sorting behavior explicit for the language section. Foundations should
usually stay ranked by relevance rather than sharing the language sort toggle.

If you need multiple sort controls, keep them minimal and legible. The simplest
acceptable version is:

- user-controlled sorting for languages
- fixed relevance ordering for foundations

### Template structure

The new template should be intentional, not a generic report dump. Avoid:

- walls of tiny metadata chips
- over-nesting
- two giant grids with no explanatory framing

Aim for a strong hierarchy:

- header
- compact foundation cards
- larger language browse grid

### Link behavior

Foundation cards must link to a sensible profile destination. If the current
profile routing does not yet handle foundations well, wire the link in the best
available way and leave the route/profile correctness cleanup to PROMPT_85.

Do not block this prompt on perfect cross-entity route cleanup.
