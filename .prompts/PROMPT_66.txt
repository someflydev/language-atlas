# PROMPT_66: LLM-Managed Dark Matter Alias & Canonical Layer

## Objective

Implement a durable, LLM-managed normalization layer for
`scripts/dark_matter_audit.py` so the dark matter audit can consolidate
semantically equivalent terms without hardcoding every case in Python.

The current audit already does useful mechanical normalization
(punctuation/casing cleanup, year stripping, etc.), but it still treats many
LLM-discernible variants as separate missing items. We now want a reviewed,
repo-managed alias system that stays deterministic at runtime while allowing
future prompt sessions to extend the mapping safely.

This session must introduce two hidden data files under `data/`:

- `data/.dark_matter_aliases.json`
- `data/.dark_matter_canonicals.json`

These files are not narrative content. They are implementation metadata used
by the dark matter audit only.

## High-Level Design

The audit should remain deterministic and code-driven, but semantic
equivalence decisions should move out of Python code and into reviewed data.

Split the concerns into two files:

### 1. `data/.dark_matter_aliases.json`

Purpose: map noisy or variant references to one reviewed canonical term,
regardless of category.

This file answers:

- "When the audit sees this phrase, what term should it really mean?"

### 2. `data/.dark_matter_canonicals.json`

Purpose: define the reviewed canonical term and what type of entity it is.

This file answers:

- "What is the normalized term?"
- "Which dark matter category should it belong to?"
- "If needed, which profile/file key should it compare against?"

Do **not** continue expanding the hardcoded alias map in Python except for
strictly mechanical rules. The goal of this prompt is to make the semantic
layer data-driven.

## Required File Locations

Create these exact hidden files:

- `data/.dark_matter_aliases.json`
- `data/.dark_matter_canonicals.json`

## Required JSON Shapes

### `data/.dark_matter_aliases.json`

Use a flat object mapping variant term to canonical term.

Example shape:

```json
{
  "shell": "Bourne Shell",
  "sh": "Bourne Shell",
  "sh (Bourne shell)": "Bourne Shell",
  "python (current state)": "Python",
  "javascript (node.js - 2009)": "JavaScript"
}
```

Rules:

- Keys are aliases or noisy terms as human-readable strings.
- Values are canonical display terms as human-readable strings.
- The loader should normalize keys and values internally using the audit's
  normalization pipeline; the file itself should stay readable.
- Alias keys must be unique after normalization.
- An alias must not point to another alias indirectly. Resolve it to the
  final canonical term directly.

### `data/.dark_matter_canonicals.json`

Use a top-level object keyed by canonical display term.

Required per-entry field:

- `type`

Optional per-entry field:

- `profile_key`

Example shape:

```json
{
  "Bourne Shell": {
    "type": "language"
  },
  "Python": {
    "type": "language"
  },
  "JavaScript": {
    "type": "language"
  },
  "Garmisch Conference": {
    "type": "historical_event",
    "profile_key": "garmisch_conference"
  }
}
```

Type must be one of:

- `language`
- `concept`
- `organization`
- `historical_event`
- `concept_combo`
- `person`

Notes:

- `profile_key` exists for cases where the canonical display term and the
  profile filename stem differ meaningfully.
- If `profile_key` is omitted, the audit should fall back to the existing
  normalization logic for comparing against files or known entities.
- Do not invent extra fields unless they are clearly needed by the final
  implementation.

## Implementation Requirements

### Part A: Refactor the Audit to Load Both Hidden Files

Update `scripts/dark_matter_audit.py` so it:

1. Loads `data/.dark_matter_aliases.json` if present.
2. Loads `data/.dark_matter_canonicals.json` if present.
3. Treats missing files as empty mappings, not fatal runtime errors.
4. Validates both files before proceeding with the audit.
5. Uses the loaded data during dark matter term resolution.

Do not bury this in ad hoc global variables if a small helper or lightweight
resolver object would make the logic clearer.

### Part B: Add Validation Rules

The audit should fail fast with a clear error message if any of the following
are true:

1. `data/.dark_matter_aliases.json` is invalid JSON.
2. `data/.dark_matter_canonicals.json` is invalid JSON.
3. An alias target does not exist as a canonical entry.
4. Two alias keys collide after normalization.
5. Two canonical keys collide after normalization.
6. A canonical entry has an unsupported `type`.
7. A `profile_key` is present but is not a string.

Make these validation failures explicit and readable. This feature is meant
to help future LLM sessions maintain the files safely.

### Part C: Introduce a Resolution Layer

Before bucketing a discovered term into `referenced_languages`,
`referenced_entities`, `referenced_organizations`,
`referenced_historical_events`, or `referenced_combos`, resolve it through
the new alias and canonical system.

The intended order is:

1. Existing mechanical cleanup:
   `strip_years`, `clean_name`, `canonicalize`, `canonicalize_event`, etc.
2. Alias lookup:
   if a noisy term maps to a canonical term, rewrite it.
3. Canonical lookup:
   if the canonical term is registered, use its declared `type`.
4. Bucketing:
   place the resolved term into the correct target map using the canonical
   display term in the report.

Important:

- The canonical file should override heuristic bucketing when there is a
  match.
- If there is no canonical entry, the existing heuristic logic should still
  work as fallback.
- Do not regress the current ability to infer organizations, historical
  events, or people heuristically.

### Part D: Compare Existing Profiles Against Canonical Terms

The audit currently compares references against existing profile files and
known data entities. Extend that logic so canonical entries can match existing
profiles correctly.

Rules:

1. If a canonical entry includes `profile_key`, compare against that.
2. Otherwise compare against the normalized canonical display term.
3. Keep the report human-readable:
   output the canonical display term, not the noisy alias that triggered it.

This is especially important for cases like:

- `Bourne Shell` represented by `Bourne_Shell.json`
- future historical events whose canonical display term differs from slug

### Part E: Remove the Current Hardcoded Semantic Alias Logic

Migrate the existing `LANGUAGE_PROFILE_ALIASES` behavior into the hidden
JSON files and remove or simplify that hardcoded Python alias block.

Mechanical normalization helpers can stay in Python.
Semantic equivalence mappings should live in the hidden data files.

If a tiny compatibility shim remains necessary, keep it minimal and document
why.

## Scope Guardrails

### Do Merge

These are the kinds of cases this system is for:

- era labels that obviously refer to an already-profiled canonical language
- shell-name variants that refer to one specific historically reviewed term
- exact duplicate spellings or punctuation variants

### Do Not Merge Automatically

Do not over-collapse broad or ambiguous terms just because an LLM can invent
a plausible relation.

Examples of terms that should stay unresolved unless explicitly reviewed:

- `AI`
- `Cloud`
- `Security`
- `Shell` if the team has not decided it should always mean `Bourne Shell`
  in a given context

For this prompt specifically, `Shell` **is** an approved alias to
`Bourne Shell` because that policy decision has already been made in the repo
discussion. Do not generalize that permissiveness to every broad term.

The tests should cover at minimum:

   `javascript (node.js - 2009)` resolves to canonical `JavaScript`

   a canonical term with a `profile_key` can match an existing profile stem

Do not write brittle tests that depend on the exact full dark matter report
line-for-line. Test the resolver behavior and the high-value audit outcomes.

This is an infrastructure prompt, not a content-expansion prompt.
