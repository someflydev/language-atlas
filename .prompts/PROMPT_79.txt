# PROMPT_79: Entity Type Classification and File Hygiene

## Objective

Introduce a formal `entity_type` classification field to the Atlas data model
and file system. The `languages` table has always held a mix of actual
programming languages, pre-computing historical foundations (Jacquard Loom,
Turing Machine, etc.), and influential tools/libraries/runtimes (LLVM, GCC,
JVM, Numpy, React, etc.). This prompt makes that distinction explicit and
machine-readable by:

1. Adding `entity_type` to the SQLite schema and populating it from source data.
2. Tagging all existing `languages.json` entries with the correct type.
3. Adding 11 new entries (10 artifacts + ENIAC) that already have profile docs
   but no `languages.json` record.
4. Resolving two profile filename issues: the orphan `PLI.json` and the
   non-ASCII `Plankalkül.json`.

Do not change routes, templates, CLI, or TUI in this prompt — surfacing
`entity_type` through the full application stack is deferred to PROMPT_81.

## Task 4: File Hygiene

### 4a. Delete the orphan PLI.json

`data/docs/language_profiles/PLI.json` is an inferior duplicate.
`data/docs/language_profiles/PL:I.json` already maps correctly to the
canonical `PL/I` entry in `languages.json` via the `:` → `/` candidate in
`DataLoader._language_profile_lookup_candidates`. Delete `PLI.json`.

```bash
rm data/docs/language_profiles/PLI.json
```

Confirm: `PL:I.json` still resolves to `PL/I` after the deletion.

### 4b. Rename Plankalkül.json to Plankalku l.json

Wait — the correct ASCII transliteration is `Plankalkul.json` (ü → u).

```bash
mv "data/docs/language_profiles/Plankalkül.json" \
   data/docs/language_profiles/Plankalkul.json
```

After the rename, `Plankalkul.json` produces the key `Plankalkul`, but the
canonical language name in `languages.json` is `Plankalkül` (with umlaut).
The existing lookup candidates (`name`, `name.replace('_',' ')`,
`name.replace(':','/')`, combined) do not strip diacritics, so the renamed
file would fail to link. Fix this by adding a diacritic-stripping candidate
to `DataLoader._language_profile_lookup_candidates` in
`src/app/core/data_loader.py`:

```python
@staticmethod
def _language_profile_lookup_candidates(name: str) -> List[str]:
    import unicodedata
    candidates: List[str] = []
    for candidate in [
        name,
        name.replace('_', ' '),
        name.replace(':', '/'),
        name.replace(':', '/').replace('_', ' '),
        unicodedata.normalize('NFKD', name).encode('ascii', 'ignore').decode('ascii'),
        unicodedata.normalize('NFKD', name.replace('_', ' ')).encode('ascii', 'ignore').decode('ascii'),
    ]:
        if candidate not in candidates:
            candidates.append(candidate)
    return candidates
```

The `unicodedata` import already exists at the top of `data_loader.py`; do
not add a duplicate import. Move the inline import to use the top-level one.
