# PROMPT_67: Expand Dark Matter Source Coverage

## Objective

Expand `scripts/dark_matter_audit.py` so it discovers dark matter from more
of the Atlas source-of-truth JSON and doc directories, rather than relying on
the current narrow subset of fields.

This prompt is a direct follow-on to `PROMPT_66`.

Assume that by the time this prompt runs, `PROMPT_66` has already been
completed and the repo already contains:

- `data/.dark_matter_aliases.json`
- `data/.dark_matter_canonicals.json`
- a refactored dark matter audit that uses those hidden files

This prompt is **not** about adding more semantic alias rules. It is about
improving source coverage so the audit notices important references that are
currently invisible or only partially visible.

The goal is simple:

- scan more of the real data corpus
- preserve deterministic behavior
- reduce false negatives
- avoid a flood of low-value garbage by expanding coverage in a ranked,
  deliberate way

## Context and Problem Statement

The current audit finds a lot of useful dark matter, but it still misses
important references because it only scans a narrow subset of the Atlas data.

Examples of current limitations:

- `data/concepts.json` is scanned only for `name` and `description`, but not
  `responsible`.
- `data/paradigms.json` is used only for `missing_paradigms`, not as a source
  of dark matter references.
- `data/eras.json` is only partially scanned; major narrative fields are
  ignored.
- `data/learning_paths.json` is not scanned at all.
- `data/docs/era_summaries/` is not scanned at all.
- `data/influences.json` is not scanned at all.

The result is that the audit currently under-reports some real missing
profiles while over-relying on prose from other places.

This prompt should correct that by expanding coverage in a controlled order.

## Design Principle

Follow this order of trust:

1. Structured source-of-truth JSON fields
2. Structured narrative JSON in `data/docs/`
3. Heuristic extraction from prose
4. Hidden alias/canonical consolidation from `PROMPT_66`

Do **not** compensate for missing field coverage by adding more alias entries.
The problem here is that the audit is not reading enough of the corpus.

## Ranked Coverage Checklist

Implement these in priority order. Keep the ranking visible in the checklist
file and in your own execution order.

### Priority 1: High-Signal Structured Fields

These are the best improvements and should be done first.

#### 1. `data/concepts.json`

Add dark matter scanning for:

- `responsible[]`

Why:

- This field directly names people or organizations associated with concepts.
- It is high signal and low noise.
- It is currently a clear blind spot.

Implementation notes:

- Each `responsible` item should be passed through the same reference-routing
  path used elsewhere.
- Respect the hidden alias/canonical layer from `PROMPT_66`.

#### 2. `data/eras.json`

Add dark matter scanning for:

- `description`
- `catalyst`
- `key_innovations[]`
- `timeline_events[].description`
- `modern_reactions[]`

Why:

- These fields contain some of the Atlas's richest historical narrative.
- They frequently mention people, events, concepts, and organizations that
  should be discoverable.
- Current coverage is too narrow and misses important references.

Implementation notes:

- Reuse existing extraction helpers where appropriate.
- Keep current scanning of `key_drivers`, `pivotal_languages`,
  `reactions_and_legacy`, and embedded `crossroads`.
- Avoid double-counting in a way that changes report semantics; deduplication
  should still happen through canonicalization and set-like maps.

#### 3. `data/docs/era_summaries/`

Add this directory to the profile-scan set.

Why:

- Era summaries are hub narratives and can mention significant entities not
  surfaced elsewhere.
- `AGENT.md` explicitly treats era summaries as important narrative files.

Implementation notes:

- Treat them similarly to other narrative JSON docs.
- Do not scan `atlas_meta`; that exclusion should remain intact.

### Priority 2: High-Value Semi-Structured Sources

These should be added after Priority 1.

#### 4. `data/paradigms.json`

Add dark matter scanning for:

- `description`
- `motivation`
- `languages[]`
- `connected_paradigms[]`
- `key_features`

Why:

- Paradigms contain real historical and conceptual references.
- `languages[]` is especially important as a structured reference source.
- `motivation` often contains named people and crises.

Implementation notes:

- Keep the existing `missing_paradigms` check.
- This prompt is about **additional** dark matter coverage, not replacing the
  current paradigm consistency logic.
- If `key_features` is a dict, scan its string values recursively.

#### 5. `data/learning_paths.json`

Add dark matter scanning for:

- `title`
- `description`
- `steps[].language`
- `steps[].milestone`
- `steps[].rationale`
- `steps[].challenge`

Why:

- Learning paths are curated Atlas content and often encode meaningful
  language and concept references.
- They are currently invisible to the audit.

Implementation notes:

- `steps[].language` should likely route directly as a language reference.
- The narrative fields should go through normal extraction.
- Be careful not to interpret every generic teaching phrase as dark matter.

### Priority 3: Lower-Risk Supplemental Sources

Add these only after the above are working and validated.

#### 6. `data/influences.json`

Consider whether this file should contribute dark matter references from:

- `from`
- `to`

Why:

- If this file contains edges not fully mirrored in `languages.json`, the
  audit should not ignore them.

Implementation notes:

- First confirm whether `influences.json` adds unique information or is fully
  redundant with `languages.json`.
- If redundant, document that and either skip it or include it harmlessly.
- Do not create noisy duplicate reporting if it adds no net value.

#### 7. `data/core_concepts.json`

Investigate this file and determine whether it should contribute references.

Why:

- It exists in `data/` and may be part of the source corpus, but it is not
  currently involved in the audit.

Implementation notes:

- If it is empty, obsolete, or not part of current Atlas runtime behavior,
  document that decision and do not force it into the audit.
- If it contains real canonical data, integrate it carefully.

## Exact Audit Behavior Requirements

### 1. Keep Existing Coverage

Do not regress current scanning of:

- `people.json`
- `languages.json`
- `concepts.json`
- `eras.json`
- narrative docs already in `profiles_dirs`

### 2. Use the PROMPT_66 Normalization Layer

Assume the alias/canonical resolution layer already exists.

Every newly added source must flow through the same normalization pipeline:

1. mechanical cleanup
2. alias resolution
3. canonical resolution
4. category bucketing

Do not reintroduce special-case semantic alias maps in Python.

### 3. Preserve Category Semantics

When scanning newly added fields:

- structured language fields should prefer language bucketing
- structured person/org fields should use normal routing and canonical rules
- prose fields should still use heuristics, but only after structured signals

If a field is obviously typed by schema, use that knowledge.

Examples:

- `learning_paths.steps[].language` is a language reference
- `concepts.responsible[]` is usually person or organization
- `paradigms.languages[]` is a language reference

### 4. Do Not Let Coverage Expansion Become Noise Explosion

This prompt is successful only if it improves recall without causing the
report to fill with junk.

Before keeping a new field in the audit, evaluate:

- does it add real missing entities?
- does it mostly add generic prose fragments?
- can the existing extraction helpers handle it acceptably?

If a field proves too noisy, document the issue and either:

- tighten extraction for that field, or
- defer it explicitly in comments and in the final response

Do not silently ship obviously bad recall/precision tradeoffs.

## Investigation Notes to Keep in Mind

These observations motivated this prompt:

- `Edsger Dijkstra` was not actually absent from dark matter; he was present
  multiple times under `missing_entities` in noisy forms
- the more serious issue is under-coverage of structured fields and some
  narrative sources

  the audit's ability to surface related names depends heavily on where it
  scans

This prompt should improve recall first. Name consolidation should remain the
job of the `PROMPT_66` hidden alias/canonical layer.
