# PROMPT_80: Name Mismatches, Missing Language Entries, and Profile Filename Fixes

## Objective

After PROMPT_79 established the `entity_type` classification and added
artifact/foundation entries, the audit still reports ~80 unknown references in
`influenced` and `influenced_by` fields. This prompt resolves the structural
causes:

1. Canonical name mismatches — the same language is referenced under different
   spellings (e.g., `C#` vs `CSharp`, `Simula 67` vs `SIMULA 67`).
2. Missing `languages.json` entries — profile docs exist in
   `data/docs/language_profiles/` but the language was never added to the
   source JSON, so influence edges are silently dropped.
3. Profile filename mismatches — a profile doc exists but its filename does
   not resolve to the canonical language name through the lookup logic.

Do not add new profile documents in this prompt. Only fix data records,
influence references, and filenames. Surfacing `entity_type` through the
application stack is deferred to PROMPT_81.

## Task 3: Fix Profile Filename Mismatches

### 3a. FLOW-MATIC profile

The file `data/docs/language_profiles/FLOW_MATIC.json` uses an underscore.
The canonical language name is `FLOW-MATIC` (with a hyphen). The lookup
candidates function does not replace underscores with hyphens, so the
profile is silently dropped during build. Rename the file:

```bash
mv data/docs/language_profiles/FLOW_MATIC.json \
   "data/docs/language_profiles/FLOW-MATIC.json"
```

Verify the profile loads for FLOW-MATIC after rebuild.

### 3b. VB.NET profile

The file `data/docs/language_profiles/VBNET.json` does not match `VB.NET`.
Rename to use the canonical name directly (the dot is valid in filenames and
is not in the URL-unsafe character set for the lookup logic):

```bash
mv data/docs/language_profiles/VBNET.json \
   "data/docs/language_profiles/VB.NET.json"
```

### 3c. F# profile and canonical name

The file `data/docs/language_profiles/FSharp.json` does not match `F#`
because `#` fails the URL-safety check in `get_entity_link_map` and also
breaks browser URL parsing at the fragment boundary.

The correct fix is to change the canonical `name` in `languages.json` from
`"F#"` to `"FSharp"` and set `"display_name": "F#"` so the human-readable
name is preserved everywhere. The route becomes `/language/FSharp`.

Steps:
1. In `languages.json`, find the `F#` entry and change:
   - `"name": "F#"` → `"name": "FSharp"`
   - Set `"display_name": "F#"` if not already present.
2. In `languages.json`, update all `influenced`/`influenced_by` references
   from `"F#"` to `"FSharp"` throughout the file.
3. `FSharp.json` already has the correct filename for lookup — no rename needed.

After this change, `get_entity_link_map` will include `FSharp` (URL-safe)
AND `F#` (display_name, also URL-safe per the `_url_safe_segment` check —
wait, `F#` contains `#` which IS in the unsafe set). The display_name `F#`
will be skipped for auto-linking, which is the correct behaviour. The
canonical route slug `FSharp` will be auto-linked instead.

import sqlite3
conn = sqlite3.connect('language_atlas.sqlite')
conn.row_factory = sqlite3.Row
for name in ('ALGOL 58', 'Bourne Shell', 'D', 'Delphi', 'PureScript',
             'FSharp', 'FLOW-MATIC', 'VB.NET', 'QUEL', 'Visual Basic',
             'Zsh', 'CSS', 'HTML'):

        "SELECT l.name, l.entity_type, lp.title FROM languages l "
        "LEFT JOIN language_profiles lp ON l.id=lp.language_id "
        "WHERE lower(l.name)=lower(?)",
        (name,)
    ).fetchone()
    if row:

    else:

conn.close()
